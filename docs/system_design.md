System Design

1. Problem Definition

Financial fraud detection is not only a binary classification problem. In real-world financial institutions, fraud detection systems must support investigation, explanation, monitoring, human review and governance.

A simple model that only predicts fraud or not fraud is not enough for practical use. Analysts need to know why a transaction is risky, what evidence supports the prediction, how confident the system is, and whether the case should be approved, reviewed or blocked.

FraudLens AI is designed as a decision-support platform for fraud investigation.

2. System Overview

The system contains the following main components:

1. Data ingestion and data lake
2. Data quality and feature engineering pipeline
3. Hybrid fraud detection engine
4. Explanation and uncertainty module
5. Drift and robustness monitor
6. Fraud investigation API
7. Analyst investigation workspace
8. Case management system
9. 3D fraud network investigation view
10. AI investigation copilot
11. Model governance and audit logging

3. Data Layer

The first version uses the PaySim synthetic financial transaction dataset.

The data layer follows a simplified medallion architecture:

* Raw layer: original downloaded dataset
* Bronze layer: schema-validated transaction data
* Silver layer: cleaned and feature-enriched transaction data
* Gold layer: model-ready training and inference tables

Planned data folders:

data/
├── raw/
├── bronze/
├── silver/
└── gold/

The raw dataset will not be uploaded to GitHub. Users should download the dataset manually and place it under data/raw/.

4. Feature Pipeline

The feature pipeline converts raw transaction records into model-ready features.

Initial features may include:

* transaction type
* transaction amount
* sender original balance
* sender new balance
* receiver original balance
* receiver new balance
* balance difference features
* transaction amount ratio
* account-level transaction frequency
* account-level total outgoing amount
* account-level total incoming amount
* time-window behavioural features
* historical fraud exposure features

Future graph features may include:

* node degree
* number of connected accounts
* number of shared devices
* number of shared IP addresses
* shortest path to known risky accounts
* community risk score
* connection to suspicious transaction clusters

5. Hybrid Risk Decision Engine

The fraud detection engine should not rely on only one model.

The planned decision engine combines:

* business rules
* supervised machine learning
* anomaly detection
* graph-based features
* probability calibration
* uncertainty estimation

The system output should include:

* transaction ID
* risk score
* uncertainty score
* recommended action
* top contributing factors
* model version
* explanation summary

Example output:

{
  "transaction_id": "TX123",
  "risk_score": 0.91,
  "uncertainty": 0.18,
  "recommended_action": "review",
  "top_factors": [
    "large transaction amount",
    "abnormal sender balance change",
    "high-risk account connection"
  ],
  "model_version": "xgboost_v1.0"
}

6. Recommended Actions

The system should recommend one of three actions:

* approve: low-risk transaction
* review: uncertain or medium-risk transaction requiring analyst review
* block: high-risk transaction requiring urgent attention

The final decision should be made by a human analyst in high-risk or uncertain cases.

7. Explainability Module

The explainability module helps analysts understand model predictions.

Initial explanation methods:

* SHAP feature attribution
* top contributing factors
* comparison with similar historical cases
* explanation summary for analysts

Future explanation methods:

* counterfactual explanations
* graph-based explanation
* natural language explanation generated by AI copilot

The explanation should clearly distinguish:

* model-based evidence
* business-rule evidence
* historical-case evidence
* AI-generated interpretation

8. Human-in-the-Loop Workflow

The analyst workflow includes:

1. The system receives a transaction.
2. The model produces a risk score.
3. The system generates an explanation.
4. High-risk or uncertain cases are sent to review.
5. The analyst opens the case.
6. The analyst checks transaction details, explanation and related accounts.
7. The analyst records investigation notes.
8. The analyst marks the case as confirmed fraud or false positive.
9. The feedback is stored for future model improvement.

This workflow reflects responsible AI principles because the model supports human judgement instead of replacing it.

9. Case Management

The case management module should support:

* case creation
* case assignment
* case status tracking
* analyst notes
* decision history
* model version record
* explanation record
* audit trail
* confirmed fraud / false positive label
* feedback for retraining

Planned case statuses:

* open
* under review
* escalated
* confirmed fraud
* false positive
* closed

10. Drift and Robustness Monitoring

The monitoring module should detect changes in data and model behaviour over time.

Planned monitoring areas:

* data quality
* feature distribution drift
* model performance drift
* prediction distribution drift
* feature attribution drift
* calibration drift
* latency and cost monitoring

This module connects the engineering system with the research goal of studying robustness under distribution shift.

11. 3D Fraud Network Investigation View

The 3D network view is designed for relationship exploration, not decorative visualisation.

Nodes may represent:

* accounts
* transactions
* devices
* IP addresses
* receivers
* merchants

Edges may represent:

* money transfer
* shared device
* shared IP address
* repeated counterparty
* temporal connection
* suspicious relationship

Visual encodings may include:

* node colour: risk level
* node size: transaction amount or accumulated risk
* edge thickness: transaction value or relationship strength
* time slider: fraud network development over time
* clustering: potential fraud ring

The goal is to help analysts discover suspicious relationship clusters and possible fraud rings.

12. AI Investigation Copilot

The AI investigation copilot should support analyst questions such as:

* Why is this transaction high risk?
* Which accounts are related to this case?
* What are the top contributing factors?
* Is the conclusion based on model output, business rules or historical cases?
* Generate a case summary for senior analyst review.
* What are the known limitations of the current model?

The copilot should use controlled tools, such as:

* get_transaction_details
* get_model_explanation
* search_related_accounts
* compare_similar_cases
* retrieve_policy_document
* generate_case_summary

The copilot must not independently approve, reject or block transactions. Human verification is required.

13. Governance and Security

The system should include:

* model card
* data card
* audit log
* role-based access control
* model version tracking
* data version tracking
* explanation record
* decision history
* rollback mechanism
* sensitive data masking
* prompt injection defence for AI copilot
* analyst verification warning

A clear warning should be shown for AI-generated outputs:

AI-generated content. Analyst verification required.

14. Summary

FraudLens AI is designed as a realistic fraud investigation platform. Its value comes from combining machine learning, explainability, human review, monitoring, governance and investigation workflow.

The project is not just a classification model. It is a responsible AI decision-support system for financial fraud investigation.
